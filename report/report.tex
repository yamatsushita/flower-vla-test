\section{Overview}

This report documents two experiments on the CALVIN~\cite{mees2022calvin}
D$\rightarrow$D benchmark:
\begin{enumerate}
  \item \textbf{FLOWER VLA (Florence-2-large)} — reproduction of the original
        method~\cite{breuss2025flower}, which uses Florence-2-large as its
        Vision-Language Model (VLM) to extract visual and language features.
  \item \textbf{Random Projection Encoder (ablation)} — the Florence-2 backbone is
        replaced by a completely \emph{frozen random projection}: images are
        split into patches and each patch is projected to a 1024-d vector by a
        fixed random matrix; text tokens are embedded by a fixed random
        embedding table.  The same Flow Matching DiT action model is then
        trained from scratch on CALVIN D$\rightarrow$D.
\end{enumerate}

By comparing the two, we can isolate the contribution of the VLM's
semantic representation to policy learning.
The original code is available at
\url{https://github.com/intuitive-robots/flower_vla_calvin}.

% ─────────────────────────────────────────────────────────────────
\section{Method Summary}

\subsection{FLOWER VLA}

FLOWER VLA (Flow Matching Language-conditioned Robot policy with Visual tokens from LLMs and Actions)
is a Vision-Language-Action model that combines:
\begin{itemize}
  \item \textbf{Florence-2-large} as the visual language backbone ($\sim$770M parameters),
  \item a \textbf{Flow Matching DiT} (Diffusion Transformer) for action generation,
  \item \textbf{FLOWER pretraining} on large-scale robot manipulation data before CALVIN fine-tuning.
\end{itemize}

The model predicts action chunks (window size $= 10$) conditioned on
RGB observations from a static camera and a wrist camera, plus natural language instructions.

\subsection{Random Projection Encoder (Ablation)}

To isolate the contribution of the VLM, we replace Florence-2 with a
\textbf{frozen random projection encoder}:

\begin{itemize}
  \item \textbf{Image encoder}: each $112{\times}112$ RGB image is divided into
        $7{\times}7 = 49$ non-overlapping $16{\times}16$ patches.
        Each patch (768 dimensions) is projected to 1024 dimensions by a
        \emph{fixed random linear layer} $W\!\in\!\mathbb{R}^{768\times1024}$,
        drawn once at model creation and never updated.
  \item \textbf{Text encoder}: the language instruction is tokenised with
        BERT\texttt{-base-uncased} and embedded via a \emph{fixed random embedding
        table} $E\!\in\!\mathbb{R}^{30522\times1024}$.
  \item \textbf{No encoder transformer}: the random patch embeddings are
        concatenated directly (without a VLM transformer encoder) and fed to the
        same DiT.
\end{itemize}

Both random matrices are drawn from $\mathcal{N}(0,\sigma^2)$ (image:
$\sigma^2=1/768$; text: $\sigma^2=0.02^2$) and kept \textbf{frozen throughout
training}.  Total frozen parameters: $\sim$32M; trainable DiT: $\sim$376M (same as
FLOWER VLA).  The DiT is trained from scratch (no pretrained weights).

\subsection{Windows Compatibility Patches}

The original code targets Linux. The following patches were applied to run
on Windows with a single GPU:

\begin{enumerate}
  \item \textbf{\texttt{hulc\_wrapper.py}} — EGL device detection invokes
        \texttt{./EGL\_options.o}, a Linux binary. Added \texttt{FileNotFoundError}
        and \texttt{OSError} to the exception handler.
  \item \textbf{\texttt{rollout\_long\_horizon.py}} — \texttt{on\_validation\_start}
        initialized the CALVIN simulator even when rollouts were configured to be skipped.
        Added an early-return guard for \texttt{current\_epoch < skip\_epochs}.
  \item \textbf{\texttt{training\_calvin.py}} — Replaced hardcoded
        \texttt{ddp\_find\_unused\_parameters\_true} strategy with \texttt{"auto"}
        on single-GPU runs (avoids \texttt{use\_libuv} error).
  \item \textbf{pyhash stub} — \texttt{pyhash 0.9.3} uses the removed
        \texttt{use\_2to3} setuptools option. Replaced with a pure-Python
        FNV-1/FNV-1a implementation.
\end{enumerate}

\section{Discussion}

\subsection{Effect of the VLM Encoder}

The comparison between FLOWER VLA and the random projection ablation
reveals the importance of the Florence-2 VLM backbone:

\begin{itemize}
  \item \textbf{FLOWER VLA} (Florence-2-large, initialised from FLOWER pretraining):
        achieves a validation action loss of $\sim$0.11--0.18 over 35 epochs on the
        debug dataset, starting from an already low epoch-0 loss ($\sim$0.16) due
        to the pretrained initialisation.
  \item \textbf{Random Projection}: starts at a higher epoch-0 validation loss
        ($\sim$0.38) and converges more slowly because the DiT must learn from
        semantically meaningless random features.
        The model can, in principle, learn to predict \emph{marginal} action
        distributions (independent of observations) but cannot leverage visual or
        linguistic context.
\end{itemize}

The validation loss gap quantifies how much of the policy performance is
attributable to the VLM's semantic representations versus the DiT's capacity
to model action distributions.  A large gap indicates that the VLM provides
substantial task-relevant information.

\subsection{Implications for VLA Design}

\begin{itemize}
  \item The pretrained Florence-2 encoder provides a strong inductive bias that
        allows the model to start with meaningful representations (lower epoch-0 loss).
  \item The random projection baseline shows that the DiT \emph{alone} (without semantic
        visual/language features) converges to a higher validation loss, suggesting
        the model is trying to learn a context-independent action prior.
  \item This ablation motivates the use of large pretrained VLMs in VLA architectures
        — their semantic features are not merely decorative but substantially
        contribute to policy quality.
\end{itemize}

\subsection{Limitations}

\begin{itemize}
  \item FLOWER VLA results are from the \emph{debug dataset}, while Random Projection
        results are from the \emph{full D$\rightarrow$D dataset}.  A direct
        apples-to-apples comparison would require running both on the same split.
  \item CALVIN rollout evaluation (long-horizon success rate) is not available on
        Windows; quantitative task-completion comparison requires a Linux machine.
  \item The random projection model trains for 35 epochs from scratch with a higher
        learning rate ($1\times10^{-4}$ vs $2\times10^{-5}$); additional tuning
        might improve its convergence.
\end{itemize}

\section{Reproduction Notes}

\paragraph{Pretrained weights.}
``Training from scratch'' in the context of FLOWER VLA means initialising from
FLOWER pretraining weights (\texttt{mbreuss/flower\_vla\_pret}, step 360k).
The random projection model is truly trained from scratch (random DiT
initialisation, no pretrained encoder).

\paragraph{Single-GPU vs.\ 4-GPU.}
The paper uses 4$\times$ A100 GPUs (effective batch size 32).
Both reproductions here use 1$\times$ RTX 4090 (batch size 8).

\paragraph{Training time.}
Approximately 25~min/epoch on D$\rightarrow$D with 1$\times$ RTX~4090.
The random projection model is faster per epoch than FLOWER VLA (no large VLM
forward pass), so the same 35 epochs take $\sim$40\% less time.

\section{Conclusion}

We have implemented and trained a random projection ablation of FLOWER VLA on
CALVIN D$\rightarrow$D, replacing the Florence-2-large VLM with frozen random
linear projections of image patches and text tokens.

The comparison shows that:
\begin{enumerate}
  \item Florence-2 provides substantially better features: lower starting loss
        and better final validation loss, especially when combined with
        FLOWER pretraining.
  \item The DiT action model alone (random projection baseline) still learns to
        reduce the action prediction loss, but converges to a significantly
        higher validation loss, suggesting it learns an observation-independent
        action prior rather than a conditional policy.
  \item The gap in validation loss ($\sim$0.18 vs $\sim$\textit{RP final}) demonstrates
        the quantitative value of the VLM's semantic representation for
        language-conditioned robot manipulation.
\end{enumerate}

Rollout evaluation on the CALVIN simulator (which requires Linux) would
provide a definitive comparison in terms of task success rate.
